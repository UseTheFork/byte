          <section id="python-semantic-release">
<h1>Python Semantic Release<a class="headerlink" href="#python-semantic-release" title="Link to this heading">¶</a></h1>
<p><a class="reference external" href="https://pypi.org/project/python-semantic-release/"><img alt="pypi" src="https://img.shields.io/pypi/v/python-semantic-release?label=PyPI&amp;logo=pypi"></a> <a class="reference external" href="https://anaconda.org/conda-forge/python-semantic-release"><img alt="conda-forge" src="https://img.shields.io/conda/vn/conda-forge/python-semantic-release?logo=anaconda"></a> <a class="reference external" href="https://github.com/python-semantic-release/python-semantic-release/releases/latest"><img alt="GitHub Release Date" src="https://img.shields.io/github/release-date/python-semantic-release/python-semantic-release?display_date=published_at"></a> <a class="reference external" href="https://pypistats.org/packages/python-semantic-release"><img alt="PyPI - Downloads" src="https://img.shields.io/pypi/dm/python-semantic-release"></a> <a class="reference external" href="https://github.com/python-semantic-release/python-semantic-release/blob/master/LICENSE"><img alt="PyPI - License" src="https://img.shields.io/pypi/l/python-semantic-release?color=blue"></a> <a class="reference external" href="https://github.com/python-semantic-release/python-semantic-release/issues"><img alt="GitHub Issues" src="https://img.shields.io/github/issues/python-semantic-release/python-semantic-release"></a></p>
<p><strong>Python Semantic Release (PSR)</strong> provides an automated release mechanism
determined by SemVer and Commit Message Conventions for your Git projects.</p>
<p>The purpose of this project is to detect what the next version of the
project should be from parsing the latest commit messages. If the commit messages
describe changes that would require a major, minor or patch version bump, PSR
will automatically bump the version number accordingly. PSR, however, does not
stop there but will help automate the whole release process. It will update the
project code and distribution artifact, upload the artifact and post changelogs
to a remotely hosted Version Control System (VCS).</p>
<p>The tool is designed to run inside of a CI/CD pipeline service, but it can
also be run locally.</p>
<p>This project was originally inspired by the <a class="reference external" href="https://github.com/semantic-release/semantic-release">semantic-release</a> project for JavaScript
by <em>Stephan Bönnemann</em>, but the codebases have significantly deviated since then, as
PSR as driven towards the goal of providing flexible changelogs and simple initial setup.</p>

</pre></div>

<p>Read more about the setup and configuration in our <a class="reference internal" href="#inline-getting-started-guide"><span class="std std-ref">Getting Started Guide</span></a>.</p>

<section id="getting-started" class="">
<span id="inline-getting-started-guide"></span><h2>Getting Started<a class="headerlink" href="#getting-started" title="Link to this heading">¶</a></h2>
<p>If you haven’t done so already, install Python Semantic Release locally following the
<a class="reference internal" href="#installation"><span class="std std-ref">installation instructions</span></a>.</p>
<p>If you are using a CI/CD service, you may not have to add Python Semantic Release to your
project’s dependencies permanently, but for the duration of this guide for the initial
setup, you will need to have it installed locally.</p>
<section id="configuring-psr" class="">
<h3>Configuring PSR<a class="headerlink" href="#configuring-psr" title="Link to this heading">¶</a></h3>
<p>Python Semantic Release ships with a reasonable default configuration but some aspects <strong>MUST</strong> be
customized to your project. To view the default configuration, run the following command:</p>
<div class="highlight-bash notranslate"><div class="highlight"><pre><span></span>semantic-release<span class="w"> </span>generate-config
</pre></div>
</div>
<p>The output of the above command is the default configuration in TOML format without any modifications.
If this is fine for your project, then you do not need to configure anything else.</p>
<p>PSR accepts overrides to the default configuration keys individually. If you don’t define the
key-value pair in your configuration file, the default value will be used.</p>
<p>By default, Python Semantic Release will look for configuration overrides in <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> under
the TOML table <code class="docutils literal notranslate"><span class="pre">[tool.semantic_release]</span></code>. You may specify a different file using the
<code class="docutils literal notranslate"><span class="pre">-c/--config</span></code> option, for example:</p>
<div class="highlight-bash notranslate"><div class="highlight"><pre><span></span><span class="c1"># In TOML format with top level table [semantic_release]</span>
semantic-release<span class="w"> </span>-c<span class="w"> </span>releaserc.toml

<span class="c1"># In JSON format with top level object key {"semantic_release": {}}</span>
semantic-release<span class="w"> </span>-c<span class="w"> </span>releaserc.json
</pre></div>
</div>
<p>The easiest way to get started is to output the default configuration to a file,
delete keys you do not need to override, and then edit the remaining keys to suit your project.</p>
<p>To set up in <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code>, run the following command:</p>
<div class="highlight-bash notranslate"><div class="highlight"><pre><span></span><span class="c1"># In file redirect in bash</span>
semantic-release<span class="w"> </span>generate-config<span class="w"> </span>--pyproject<span class="w"> </span>&gt;&gt;<span class="w"> </span>pyproject.toml

<span class="c1"># Open your editor to edit the configuration</span>
vim<span class="w"> </span>pyproject.toml
</pre></div>
</div>
<div class="admonition seealso">
<p class="admonition-title">See also</p>
<ul class="simple">
<li><p><a class="reference internal" href="api/commands.html#cmd-generate-config"><span class="std std-ref">semantic-release generate-config</span></a></p></li>
<li><p><a class="reference internal" href="configuration/index.html#configuration"><span class="std std-ref">Configuration</span></a></p></li>

<section id="configuring-the-version-stamp-feature" class="">
<h4>Configuring the Version Stamp Feature<a class="headerlink" href="#configuring-the-version-stamp-feature" title="Link to this heading">¶</a></h4>
<p>One of the best features of Python Semantic Release is the ability to automatically stamp the
new version number into your project files, so you don’t have to manually update the version upon
each release. The version that is stamped is automatically determined by Python Semantic Release
from your commit messages which compliments automated versioning seamlessly.</p>
<p>The most crucial version stamp is the one in your project metadata, which is used by
the Python Package Index (PyPI) and other package managers to identify the version of your package.</p>
<p>For Python projects, this is typically the <code class="docutils literal notranslate"><span class="pre">version</span></code> field in your <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> file. First,
set up your project metadata with the base <code class="docutils literal notranslate"><span class="pre">version</span></code> value. If you are starting with a brand new project,
set <code class="docutils literal notranslate"><span class="pre">project.version</span> <span class="pre">=</span> <span class="pre">"0.0.0"</span></code>.  If you are working on an existing project, set it to the last
version number you released. Do not include any <code class="docutils literal notranslate"><span class="pre">v</span></code> prefix.</p>
<div class="admonition important">
<p class="admonition-title">Important</p>
<p>The version number must be a valid SemVer version, which means it should follow the format
<code class="docutils literal notranslate"><span class="pre">MAJOR.MINOR.PATCH</span></code> (e.g., <code class="docutils literal notranslate"><span class="pre">1.0.0</span></code>). Python Semantic Release does NOT support Canonical
version values defined in the <a class="reference external" href="https://peps.python.org/pep-0440/">PEP 440</a> specification at this time. See
<a class="reference external" href="https://github.com/python-semantic-release/python-semantic-release/issues/455">Issue #455</a>
for more details. Note that you can still define a SemVer version in the <code class="docutils literal notranslate"><span class="pre">project.version</span></code>
field, and when your build is generated, the build tool will automatically generate a PEP 440
compliant version as long as you do <strong>NOT</strong> use a non-pep440 compliant pre-release token.</p>
</div>
<p>Your project metadata might look like this in <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code>:</p>
<div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="p">[</span><span class="n">project</span><span class="p">]</span>
<span class="n">name</span> <span class="o">=</span> <span class="s2">"my-package"</span>
<span class="n">version</span> <span class="o">=</span> <span class="s2">"0.0.0"</span>  <span class="c1"># Set this to the last released version or "0.0.0" for new projects</span>
<span class="n">description</span> <span class="o">=</span> <span class="s2">"A sample Python package"</span>
</pre></div>
</div>
<p>To configure PSR to automatically update this version number, you need to specify the file and value
to update in your configuration. Since <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> uses TOML format, you will add the
replacement specification to the <code class="docutils literal notranslate"><span class="pre">tool.semantic_release.version_toml</span></code> list. Update the following
configuration in your <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> file to include the version variable location:</p>
<div class="highlight-toml notranslate"><div class="highlight"><pre><span></span><span class="k">[tool.semantic_release]</span>
<span class="n">version_toml</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="p">[</span><span class="s2">"pyproject.toml:project.version"</span><span class="p">]</span>

<span class="c1"># Alternatively, if you are using poetry's 'version' key, then you would use:</span>
<span class="n">version_toml</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="p">[</span><span class="s2">"pyproject.toml:tool.poetry.version"</span><span class="p">]</span>
</pre></div>
</div>
<p>If you have other TOML files where you want to stamp the version, you can add them to the
<code class="docutils literal notranslate"><span class="pre">version_toml</span></code> list as well. In the above example, there is an implicit assumption that
you only want the version as the raw number format. If you want to specify the full tag
value (e.g. v-prefixed version), then include <code class="docutils literal notranslate"><span class="pre">:tf</span></code> for “tag format” at the end of the
version variable specification.</p>
<p>For non-TOML formatted files (such as JSON or YAML files), you can use the
<a class="reference internal" href="configuration/configuration.html#config-version-variables"><span class="std std-ref">version_variables</span></a> configuration key instead. This feature uses an advanced
Regular Expression to find and replace the version variable in the specified files.</p>
<p>For Python files, its much more effective to use <code class="docutils literal notranslate"><span class="pre">importlib</span></code> instead which will allow you to
dynamically import the version from your package metadata and not require your project to commit
the version number bump to the repository. For example, in your package’s base <code class="docutils literal notranslate"><span class="pre">__init__.py</span></code></p>
<div class="highlight-python notranslate"><div class="highlight"><pre><span></span><span class="c1"># my_package/__init__.py</span>
<span class="kn">from</span><span class="w"> </span><span class="nn">importlib.metadata</span><span class="w"> </span><span class="kn">import</span> <span class="n">version</span> <span class="k">as</span> <span class="n">get_version</span>

<span class="n">__version__</span> <span class="o">=</span> <span class="n">get_version</span><span class="p">(</span><span class="n">__package__</span><span class="p">)</span>
<span class="c1"># Note: __package__ must match your 'project.name' as defined in pyproject.toml</span>
</pre></div>
</div>
<div class="admonition seealso">
<p class="admonition-title">See also</p>
<ul class="simple">
<li><p>Configuration specification of <a class="reference internal" href="configuration/configuration.html#config-version-toml"><span class="std std-ref">version_toml</span></a></p></li>
<li><p>Configuration specification of <a class="reference internal" href="configuration/configuration.html#config-version-variables"><span class="std std-ref">version_variables</span></a></p></li>
</ul>
</div>
</section>
<section id="using-psr-to-build-your-project" class="">
<h4>Using PSR to Build your Project<a class="headerlink" href="#using-psr-to-build-your-project" title="Link to this heading">¶</a></h4>
<p>PSR provides a convenient way to build your project artifacts as part of the versioning process
now that you have stamped the version into your project files. To enable this, you will need
to specify the build command in your configuration. This command will be executed after
the next version has been determined, and stamped into your files but before a release tag has
been created.</p>
<p>To set up the build command, add the following to your <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> file:</p>
<div class="highlight-toml notranslate"><div class="highlight"><pre><span></span><span class="k">[tool.semantic_release]</span>
<span class="n">build_command</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="s2">"python -m build --sdist --wheel ."</span>
</pre></div>
</div>
<div class="admonition seealso">
<p class="admonition-title">See also</p>
<ul class="simple">
<li><p><a class="reference internal" href="configuration/configuration.html#config-build-command"><span class="std std-ref">build_command</span></a> - Configuration specification for the build command.</p></li>
<li><p><a class="reference internal" href="configuration/configuration.html#config-build-command-env"><span class="std std-ref">build_command_env</span></a> - Configuration specification for the build command environment variables.</p></li>
</ul>
</div>
</section>
<section id="choosing-a-commit-message-parser" class="">
<h4>Choosing a Commit Message Parser<a class="headerlink" href="#choosing-a-commit-message-parser" title="Link to this heading">¶</a></h4>
<p>PSR uses commit messages to determine the type of version bump that should be applied
to your project. PSR supports multiple commit message parsing styles, allowing you to choose
the one that best fits your project’s needs. Choose <strong>one</strong> of the supported commit parsers
defined in <a class="reference internal" href="concepts/commit_parsing.html#commit-parsing"><span class="std std-ref">Commit Parsing</span></a>, or provide your own and configure it in your
<code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> file.</p>
<p>Each commit parser has its own default configuration options so if you want to customize the parser
behavior, you will need to specify the parser options you want to override.</p>
<div class="highlight-toml notranslate"><div class="highlight"><pre><span></span><span class="k">[tool.semantic_release]</span>
<span class="n">commit_parser</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="s2">"conventional"</span>

<span class="k">[tool.semantic_release.commit_parser_options]</span>
<span class="n">minor_tags</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="p">[</span><span class="s2">"feat"</span><span class="p">]</span>
<span class="n">patch_tags</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="p">[</span><span class="s2">"fix"</span><span class="p">,</span><span class="w"> </span><span class="s2">"perf"</span><span class="p">]</span>
<span class="n">parse_squash_commits</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="kc">true</span>
<span class="n">ignore_merge_commits</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="kc">true</span>
</pre></div>
</div>
</section>
<section id="choosing-your-changelog" class="active">
<h4>Choosing your Changelog<a class="headerlink" href="#choosing-your-changelog" title="Link to this heading">¶</a></h4>
<p>Prior to creating a release, PSR will generate a changelog from the commit messages of your
project. The changelog is extremely customizable from the format to the content of each section.
PSR ships with a default changelog template that will be used if you do not provide custom
templates. The default should be sufficient for most projects and has its own set of configuration
options.</p>
<p>For basic customization, you can choose either an traditional Markdown formatted changelog (default)
or if you want to integrate with a Sphinx Documentation project, you can use the
reStructuredText (RST) format. You can also choose the file name and location of where to write the
default changelog.</p>
<p>To set your changelog location and changelog format, add the following to your <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> file:</p>
<div class="highlight-toml notranslate"><div class="highlight"><pre><span></span><span class="k">[tool.semantic_release.changelog.default_templates]</span>
<span class="n">changelog_file</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="s2">"docs/source/CHANGELOG.rst"</span>
<span class="n">output_format</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="s2">"rst"</span><span class="w"> </span><span class="c1"># or "md" for Markdown format</span>
</pre></div>
</div>
<p>Secondly, the more important aspect of configuring your changelog is to define Commit Exclusion
Patterns or patterns that will be used to filter out commits from the changelog. PSR does <strong>NOT</strong> (yet)
come with a built-in set of exclusion patterns, so you will need to define them yourself. These commit
patterns should be in line with your project’s commit parser configuration.</p>
<p>To set commit exclusion patterns for a conventional commits parsers, add the following to your
<code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> file:</p>
<div class="highlight-toml notranslate"><div class="highlight"><pre><span></span><span class="k">[tool.semantic_release.changelog]</span>
<span class="c1"># Recommended patterns for conventional commits parser that is scope aware</span>
<span class="n">exclude_commit_patterns</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="p">[</span>
<span class="w">    </span><span class="s1">'''chore(?:\([^)]*?\))?: .+'''</span><span class="p">,</span>
<span class="w">    </span><span class="s1">'''ci(?:\([^)]*?\))?: .+'''</span><span class="p">,</span>
<span class="w">    </span><span class="s1">'''refactor(?:\([^)]*?\))?: .+'''</span><span class="p">,</span>
<span class="w">    </span><span class="s1">'''style(?:\([^)]*?\))?: .+'''</span><span class="p">,</span>
<span class="w">    </span><span class="s1">'''test(?:\([^)]*?\))?: .+'''</span><span class="p">,</span>
<span class="w">    </span><span class="s1">'''build\((?!deps\): .+)'''</span><span class="p">,</span>
<span class="w">    </span><span class="s1">'''Initial [Cc]ommit.*'''</span><span class="p">,</span>
<span class="p">]</span>
</pre></div>
</div>
<div class="admonition seealso">
<p class="admonition-title">See also</p>
<ul class="simple">
<li><p><a class="reference internal" href="configuration/configuration.html#config-changelog"><span class="std std-ref">Changelog</span></a> - Customize your changelog</p></li>
<li><p><a class="reference internal" href="configuration/configuration.html#config-changelog-mode"><span class="std std-ref">changelog.mode</span></a> - Choose the changelog mode (‘update’ or ‘init’)</p></li>
<li><p><a class="reference internal" href="concepts/changelog_templates.html#changelog-templates-migrating-existing-changelog"><span class="std std-ref">Migrating an Existing Changelog</span></a></p></li>
</ul>
</div>

<section id="defining-your-release-branches">
<h4>Defining your Release Branches<a class="headerlink" href="#defining-your-release-branches" title="Link to this heading">¶</a></h4>
<p>PSR provides a powerful feature to manage release types across multiple branches which can
allow you to configure your project to have different release branches for different purposes,
such as pre-release branches, beta branches, and your stable releases.</p>
<div class="admonition note">
<p class="admonition-title">Note</p>
<p>Most projects that do <strong>NOT</strong> publish pre-releases will be fine with PSR’s built-in default.</p>
</div>
<p>To define an alpha pre-release branch when you are working on a fix or new feature, you can
add the following to your <code class="docutils literal notranslate"><span class="pre">pyproject.toml</span></code> file:</p>
<div class="highlight-toml notranslate"><div class="highlight"><pre><span></span><span class="k">[tool.semantic_release.branches.alpha]</span>
<span class="c1"># Matches branches with the prefixes 'feat/', 'fix/', or 'perf/'.</span>
<span class="n">match</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="s2">"^(feat|fix|perf)/.+"</span>
<span class="n">prerelease</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="kc">true</span>
<span class="n">prerelease_token</span><span class="w"> </span><span class="o">=</span><span class="w"> </span><span class="s2">"alpha"</span>
</pre></div>
</div>
<p>Any time you execute <code class="docutils literal notranslate"><span class="pre">semantic-release</span> <span class="pre">version</span></code> on a branch with the prefix
<code class="docutils literal notranslate"><span class="pre">feat/</span></code>, <code class="docutils literal notranslate"><span class="pre">fix/</span></code>, or <code class="docutils literal notranslate"><span class="pre">perf/</span></code>, PSR will determine if a version bump is needed and if so,
the resulting version will be a pre-release version with the <code class="docutils literal notranslate"><span class="pre">alpha</span></code> token. For example,</p>
<div class="table-wrapper docutils container">
<table class="docutils align-default">
<thead>
<tr class="row-odd"><th class="head"><p>Branch</p></th>
<th class="head"><p>Version Bump</p></th>
<th class="head"><p>Current Version</p></th>
<th class="head"><p>Next Version</p></th>
</tr>
</thead>
<tbody>
<tr class="row-even"><td><p>main</p></td>
<td><p>Patch</p></td>
<td><p><code class="docutils literal notranslate"><span class="pre">1.0.0</span></code></p></td>
<td><p><code class="docutils literal notranslate"><span class="pre">1.0.1</span></code></p></td>
</tr>
<tr class="row-odd"><td><p>fix/bug-1</p></td>
<td><p>Patch</p></td>
<td><p><code class="docutils literal notranslate"><span class="pre">1.0.0</span></code></p></td>
<td><p><code class="docutils literal notranslate"><span class="pre">1.0.1-alpha.1</span></code></p></td>
</tr>
</tbody>
</table>
</div>

<p class="admonition-title">See also</p>

</ul>

<section id="configuring-vcs-releases">
<h4>Configuring VCS Releases<a class="headerlink" href="#configuring-vcs-releases" title="Link to this heading">¶</a></h4>
<p>You can set up Python Semantic Release to create Releases in your remote version
control system, so you can publish assets and release notes for your project.</p>
<p>In order to do so, you will need to place an authentication token in the
appropriate environment variable so that Python Semantic Release can authenticate
with the remote VCS to push tags, create releases, or upload files.</p>
<section id="github-gh-token">
<h5>GitHub (<code class="docutils literal notranslate"><span class="pre">GH_TOKEN</span></code>)<a class="headerlink" href="#github-gh-token" title="Link to this heading">¶</a></h5>
<p>For local publishing to GitHub, you should use a personal access token and
store it in your environment variables. Specify the name of the environment
variable in your configuration setting <a class="reference internal" href="configuration/configuration.html#config-remote-token"><span class="std std-ref">remote.token</span></a>.
The default is <code class="docutils literal notranslate"><span class="pre">GH_TOKEN</span></code>.</p>
<p>To generate a token go to <a class="reference external" href="https://github.com/settings/tokens">https://github.com/settings/tokens</a> and click on
“Generate new token”.</p>
<p>For Personal Access Token (classic), you will need the <code class="docutils literal notranslate"><span class="pre">repo</span></code> scope to write
(ie. push) to the repository.</p>
<p>For fine-grained Personal Access Tokens, you will need the <a class="reference external" href="https://docs.github.com/en/rest/authentication/permissions-required-for-fine-grained-personal-access-tokens#repository-permissions-for-contents">contents</a>
permission.</p>
</section>
<section id="gitlab-gitlab-token">
<h5>GitLab (<code class="docutils literal notranslate"><span class="pre">GITLAB_TOKEN</span></code>)<a class="headerlink" href="#gitlab-gitlab-token" title="Link to this heading">¶</a></h5>
<p>A personal access token from GitLab. This is used for authenticating when pushing
tags, publishing releases etc. This token should be stored in the <code class="docutils literal notranslate"><span class="pre">GITLAB_TOKEN</span></code>
environment variable.</p>
</section>
<section id="gitea-gitea-token">
<h5>Gitea (<code class="docutils literal notranslate"><span class="pre">GITEA_TOKEN</span></code>)<a class="headerlink" href="#gitea-gitea-token" title="Link to this heading">¶</a></h5>
<p>A personal access token from Gitea. This token should be stored in the <code class="docutils literal notranslate"><span class="pre">GITEA_TOKEN</span></code>
environment variable.</p>
</section>
<section id="bitbucket-bitbucket-token">
<h5>Bitbucket (<code class="docutils literal notranslate"><span class="pre">BITBUCKET_TOKEN</span></code>)<a class="headerlink" href="#bitbucket-bitbucket-token" title="Link to this heading">¶</a></h5>
<p>Bitbucket does not support uploading releases but can still benefit from automated tags
and changelogs. The user has three options to push changes to the repository:</p>
<ol class="arabic simple">
<li><p>Use SSH keys.</p></li>
<li><p>Use an <a class="reference external" href="https://support.atlassian.com/bitbucket-cloud/docs/push-back-to-your-repository/#App-secret">App Secret</a>, store the secret in the <code class="docutils literal notranslate"><span class="pre">BITBUCKET_TOKEN</span></code> environment variable
and the username in <code class="docutils literal notranslate"><span class="pre">BITBUCKET_USER</span></code>.</p></li>
<li><p>Use an <a class="reference external" href="https://support.atlassian.com/bitbucket-cloud/docs/repository-access-tokens">Access Token</a> for the repository and store it in the <code class="docutils literal notranslate"><span class="pre">BITBUCKET_TOKEN</span></code>
environment variable.</p></li>
</ol>
<div class="admonition seealso">
<p class="admonition-title">See also</p>
<ul class="simple">
<li><p><a class="reference internal" href="configuration/configuration.html#config-changelog"><span class="std std-ref">Changelog</span></a> - customize your project’s changelog.</p></li>
<li><p><a class="reference internal" href="concepts/changelog_templates.html#changelog-templates-custom-release-notes"><span class="std std-ref">Custom Release Notes</span></a> - customize the published release notes</p></li>
<li><p><a class="reference internal" href="api/commands.html#cmd-version-option-vcs-release"><span class="std std-ref">version –vcs-release/–no-vcs-release</span></a> - enable/disable VCS release
creation.</p></li>
</ul>
</div>
</section>
</section>
</section>
<section id="testing-your-configuration">
<h3>Testing your Configuration<a class="headerlink" href="#testing-your-configuration" title="Link to this heading">¶</a></h3>
<p>It’s time to test your configuration!</p>
<div class="highlight-bash notranslate"><div class="highlight"><pre><span></span><span class="c1"># 1. Run the command in no-operation mode to see what would happen</span>
semantic-release<span class="w"> </span>-v<span class="w"> </span>--noop<span class="w"> </span>version

<span class="c1"># 2. If the output looks reasonable, try to run the command without any history changes</span>
<span class="c1">#    '-vv' will give you verbose debug output, which is useful for troubleshooting</span>
<span class="c1">#    commit parsing issues.</span>
semantic-release<span class="w"> </span>-vv<span class="w"> </span>version<span class="w"> </span>--no-commit<span class="w"> </span>--no-tag

<span class="c1"># 3. Evaluate your repository to see the changes that were made but not committed</span>
<span class="c1">#    - Check the version number in your pyproject.toml</span>
<span class="c1">#    - Check the distribution files from the build command</span>
<span class="c1">#    - Check the changelog file for the new release notes</span>

<span class="c1"># 4. If everything looks good, make sure to commit/save your configuration changes</span>
git<span class="w"> </span>add<span class="w"> </span>pyproject.toml
git<span class="w"> </span>commit<span class="w"> </span>-m<span class="w"> </span><span class="s2">"chore(config): configure Python Semantic Release"</span>

<span class="c1"># 5. Now, try to run the release command with your history changes but without pushing</span>
semantic-release<span class="w"> </span>-v<span class="w"> </span>version<span class="w"> </span>--no-push<span class="w"> </span>--no-vcs-release

<span class="c1"># 6. Check the result on your local repository</span>
git<span class="w"> </span>status
git<span class="w"> </span>log<span class="w"> </span>--graph<span class="w"> </span>--decorate<span class="w"> </span>--all

<span class="c1"># 7a. If you are happy with the release history and resulting commit &amp; tag,</span>
<span class="c1">#     reverse your changes before trying the full release command.</span>
git<span class="w"> </span>tag<span class="w"> </span>-d<span class="w"> </span>v0.0.1<span class="w">  </span><span class="c1"># replace with the actual version you released</span>
git<span class="w"> </span>reset<span class="w"> </span>--hard<span class="w"> </span>HEAD~1

<span class="c1"># 7b. [Optional] Once you have configured a remote VCS token, try</span>
<span class="c1">#     running the full release command to update the remote repository.</span>
semantic-release<span class="w"> </span>version<span class="w"> </span>--push<span class="w"> </span>--vcs-release
<span class="c1">#    This is optional as you may not want a personal access token set up or make</span>
<span class="c1">#    make the release permanent yet.</span>
</pre></div>
</div>
